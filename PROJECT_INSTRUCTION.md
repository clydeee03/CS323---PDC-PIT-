# PROJECT INSTRUCTION
## Distributed Multiplayer Tetris (Parallel and Distributed Computing)

### 1) Purpose and Scope
Build a research-oriented LAN multiplayer Tetris platform that demonstrates distributed systems and parallel computing principles, not just gameplay.

The system must prove:
- Deterministic simulation across distributed nodes
- Lockstep input synchronization
- Concurrent execution across dedicated threads
- Fault tolerance under LAN instability
- Measurable performance with repeatable experiments

### 2) Technology Stack (Required)
- `C++20` (or newer supported modern C++)
- `SDL2` for rendering/input windowing
- `ENet` for UDP-based networking
- `std::thread` + `std::mutex` + `std::condition_variable` + atomics
- Host-authoritative LAN architecture with deterministic lockstep

### 3) System Architecture (Layered and Modular)
Implement the system as separate modules with explicit interfaces.

#### 3.1 Presentation Layer (`render/`)
Responsibilities:
- Board and tetromino rendering
- HUD: score, combo, attack, match state
- Diagnostic overlay: FPS, RTT, packet stats, sync status

Constraints:
- Rendering must be decoupled from simulation tick rate
- Double buffering enabled
- Only render from immutable frame snapshots or safely synchronized state views

#### 3.2 Simulation Layer (`simulation/`)
Responsibilities:
- Tetromino spawn, move, rotate, collision, lock, clear
- Gravity, soft/hard drop, hold, queue
- Garbage application and victory conditions

Constraints:
- Fixed timestep simulation (`20-30` ticks/sec target, configurable)
- Deterministic behavior: same seed + same ordered inputs => identical state
- No non-deterministic time sources in simulation logic

#### 3.3 Synchronization Layer (`synchronization/`)
Responsibilities:
- Lockstep frame coordinator
- Input buffering per player per frame
- Frame gate (simulate only when required inputs are available)
- Periodic state hash generation and desync detection
- Rollback hook interfaces (stub-ready for future extension)

Constraints:
- Input is frame-indexed and validated
- Missing input handling policy must be explicit (wait/timeout/drop)
- Desync reports must include frame, hash, and node identifiers

#### 3.4 Networking Layer (`network/`)
Responsibilities:
- ENet host/client setup and lifecycle
- Packet encode/decode and versioning
- Reliable/unreliable channel policy
- Lobby join/leave, heartbeat, timeouts, RTT tracking

Required packet types:
- `INPUT_PACKET`
- `MATCH_START_PACKET`
- `PLAYER_JOIN_PACKET`
- `PLAYER_LEAVE_PACKET`
- `GARBAGE_PACKET`
- `HEARTBEAT_PACKET`
- `STATE_HASH_PACKET`
- `MATCH_END_PACKET`

Constraints:
- Primary synchronization method is input sync (not full board sync)
- Full-state sync allowed only as experimental comparison mode

#### 3.5 System Layer (`core/`, `threading/`, `metrics/`, `utils/`)
Responsibilities:
- Thread lifecycle and shutdown orchestration
- Clock/timer abstractions
- Structured logging and trace events
- Metrics collection and export

### 4) Distributed Computing Requirements
Each running game instance is an independent node.

Must demonstrate:
- Message-passing coordination only (no shared memory across nodes)
- Host-authoritative control for match setup, seed, arbitration
- Local independent simulation on each node
- Global consistency checks via frame hashes

Packet metadata minimum:
- Frame number
- Player ID
- Timestamp
- Payload type
- Payload data

### 5) Parallel Computing Requirements
Run at least these concurrent threads:

1. `Render Thread`
- SDL event poll + render submit/present

2. `Simulation Thread`
- Fixed-rate frame advancement and game rules

3. `Network Thread`
- ENet service loop, packet RX/TX, queue handoff

4. `Metrics Thread`
- Runtime telemetry aggregation and periodic logging

Concurrency requirements:
- Use producer-consumer queues between network/simulation
- Protect shared resources via mutex/CV or lock-free structures where safe
- Use atomics for counters and run-state flags
- Include clean startup/shutdown barrier logic

### 6) Deterministic Lockstep Protocol (Core)
For frame `F`:
1. Collect local input for `F`
2. Broadcast local input packet for `F`
3. Receive remote inputs for `F`
4. Wait until required inputs are present (or timeout policy triggers)
5. Simulate frame `F`
6. Every `N` frames, publish/compare state hash
7. Advance to `F+1`

Rules:
- Only input commands are synchronized for gameplay
- Shared seed is generated by host and distributed before match start
- All PRNG usage in simulation must come from deterministic seeded source

Suggested input command set:
- `MOVE_LEFT`, `MOVE_RIGHT`, `ROTATE_CW`, `ROTATE_CCW`
- `SOFT_DROP`, `HARD_DROP`, `HOLD`

### 7) Fault Tolerance and Recovery Behavior
Required mechanisms:
- Heartbeat and timeout detection
- Graceful disconnect handling
- Late/missing packet handling with policy
- Session termination protocol if quorum/game cannot continue

Minimum behavior on failures:
- Log incident with timestamps and node/player IDs
- Preserve consistency by pausing or ending match deterministically
- Avoid silent divergence

### 8) Performance Instrumentation and Analysis
Collect at runtime:
- FPS
- Simulation TPS (ticks/sec)
- CPU utilization
- Memory usage
- RTT/latency
- Packet throughput (pps, bytes/sec)
- Packet loss estimate
- Input delay distribution
- Desync frequency
- Frame stall counts

Logging requirements:
- Connection and disconnection events
- Packet send/receive summaries
- Sync hash checks and mismatch reports
- Thread loop health metrics

### 9) Experimental Comparison (Mandatory)
Implement two switchable networking modes:

A. `Input Sync Mode` (primary, lockstep)  
B. `Full-State Sync Mode` (experimental baseline only)

Compare at minimum:
- Bandwidth usage
- CPU load
- Mean/95th percentile latency
- Sync accuracy / divergence rate

Deliver results as repeatable benchmark runs on the same LAN setup.

### 10) Codebase Organization (Required)
Use this structure:

```text
/src
  /core
  /render
  /network
  /simulation
  /threading
  /metrics
  /synchronization
  /utils
/include
/tests
/assets
/logs
/docs
```

Add:
- `docs/architecture.md`
- `docs/protocol.md`
- `docs/experiments.md`

### 11) Engineering Standards
- Prefer RAII and deterministic ownership
- Avoid global mutable state
- Use smart pointers and value semantics where appropriate
- Define interfaces between layers to avoid coupling
- Keep render and simulation clocks independent
- Use compile-time warnings and static analysis in CI

### 12) Non-Functional Constraints
Must not:
- Use full-state sync as primary architecture
- Block render loop on network calls
- Tie simulation speed to render FPS
- Use unsynchronized local RNG across clients
- Hide synchronization errors

### 13) Development Phases and Exit Criteria
#### Phase 1: Deterministic Singleplayer
Exit: Same input replay yields identical hashes.

#### Phase 2: Fixed Timestep Engine
Exit: Stable TPS under variable FPS.

#### Phase 3: ENet Transport Foundation
Exit: Host/client connect, packet roundtrip validated.

#### Phase 4: LAN Match Orchestration
Exit: Join/leave/start/end flow complete.

#### Phase 5: Lockstep Input Sync
Exit: Multi-node simulation advances in frame lock.

#### Phase 6: Multithreading Integration
Exit: Dedicated threads running with safe queues and shutdown.

#### Phase 7: State Hash Validation
Exit: Periodic hash checks and desync alarms working.

#### Phase 8: Metrics and Logging
Exit: Runtime telemetry persisted and viewable.

#### Phase 9: Fault Tolerance
Exit: Timeout/disconnect scenarios handled gracefully.

#### Phase 10: Comparative Experiments
Exit: Input-sync vs full-state-sync report completed.

### 14) Test and Validation Requirements
Minimum tests:
- Deterministic replay tests
- Packet serialization/deserialization tests
- Queue/thread safety tests
- Hash consistency tests
- Disconnect/timeout integration tests

Validation scenario set:
- `2-4` LAN nodes
- Normal play under stable network
- Simulated packet loss/latency/jitter
- Node dropout mid-match

### 15) Deliverables
1. Executable host/client multiplayer system
2. Source code with required modular structure
3. Protocol and architecture documentation
4. Reproducible benchmark results and analysis
5. Logs demonstrating synchronization and fault handling

### 16) Success Criteria (Final)
The project is accepted when it demonstrates:
- Distributed execution across LAN nodes
- Deterministic lockstep synchronization using input-only gameplay sync
- Parallel processing with dedicated functional threads
- Robust failure handling and graceful degradation
- Quantitative performance evidence and comparison study

This implementation must read as a distributed systems and parallel computing artifact that uses Tetris as the experimental workload.
